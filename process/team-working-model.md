# Team working model

This document describes how the engineering team works across projects.

## Roles

### Software engineering manager / full-stack architect

Responsible for technical direction, architecture support, engineering standards, delivery health, and reducing technical risk. May also take Developers responsibilities as needed. 

### Developers

Responsible for implementing, testing, reviewing, documenting, and maintaining the systems they work on.

### Product ownership

Product ownership may be shared by business, operations, design, and project leadership.

Product context and business decisions live in Google Workspace. Technical decisions must be reflected in engineering documentation when they affect implementation.

### Project management

Project management owns delivery coordination, visibility, sequencing, and communication.

Jira is used to track work. Documentation is used to preserve knowledge.

## Team expectations

We work in a way that reduces dependency on individual memory.

That means:

- decisions are written down
- setup instructions are clear and maintained
- deployment and rollback steps are documented
- project-specific exceptions are visible
- new developers can onboard without relying only on meetings

# Asynchronous collaboration

The team operates across time zones (Canada and Cambodia).  
Asynchronous work is the default mode of collaboration.

This means work should not depend on immediate responses.

### Core principles

- Work should be able to progress without real-time interaction
- Decisions, context, and outcomes must be written down
- Communication should be clear, complete, and actionable
- Questions should include enough context to be answered without back-and-forth

### Communication expectations

When asking a question:

- include context (what you are trying to achieve)
- include what you already tried
- include relevant links (Jira, code, docs)
- be explicit about what you need (decision, review, clarification)

Avoid:

- vague questions without context
- messages that require multiple follow-ups to understand the issue

Prefer:

> “I’m implementing X in project Y. I tried approach A and B but ran into issue Z.  
> I think option A is better because of X.  
> Can you confirm or suggest a better approach?”

### Documentation as default

If something is important, it must be written down.

This includes:

- decisions
- setup instructions
- deployment steps
- known issues
- workarounds

Documentation replaces repeated explanations and reduces dependency on overlapping work hours.

### Handoffs

Work should be handed off in a way that allows progress in the next time zone.

A good handoff includes:

- current status
- what was completed
- what remains
- known issues or blockers
- next recommended steps

### Response expectations

Responses are not expected to be immediate.

Each team member should:

- communicate blockers clearly
- provide updates when work cannot proceed
- prioritize clarity over speed

### Synchronous time (when needed)

Synchronous meetings should be used intentionally for:

- complex discussions
- alignment on important decisions
- onboarding or knowledge transfer

They should not be required for routine progress.

## Knowledge capture

If a question is asked more than once, the answer must be documented.

Documentation should be updated in the appropriate place:

- global documentation for shared standards and processes
- project documentation for project-specific behavior

The goal is to reduce repeated explanations and make knowledge accessible without relying on individuals.

When answering a repeated question:

1. Answer the question
2. Add or update the documentation
3. Share the link to the documentation instead of rewriting the answer next time :)
